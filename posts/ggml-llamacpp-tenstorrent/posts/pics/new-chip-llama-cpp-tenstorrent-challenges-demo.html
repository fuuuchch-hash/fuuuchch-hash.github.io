管理层可以带走的结论

接入层不是主要工作量

注册 backend、接 CMake、暴露 device API 是入口。真正的工作在接口背后的内存和算子语义。

布局不一致会放大成本

GGML 常按 row-major、stride、CPU 可寻址来理解 tensor；Tenstorrent 侧以 tiled tensor 为核心，这会影响 copy、view、fallback 和调度。

算子覆盖决定能否稳定推理

只让第一层跑起来是里程碑，但要支撑真实模型，需要 ROPE、MUL_MAT、GET_ROWS、SOFT_MAX、VIEW、broadcast、KV cache 等链条一起成立。

生产化还包括维护成本

TTNN/Metalium 版本、JIT 冷启动、多卡 mesh、精度转换、上游接口变化都会持续消耗工程资源。

Hosttokens、batch、prompt、KV 状态仍由 CPU 组织。
GGML Graph算子节点看起来是通用 tensor 运算。
Scheduler依据 backend 能力切分 CPU / Metalium 子图。
Layout Bridgerow-major tensor 需要转换成 TTNN tile layout。
TTNN / Metalium优先调 TTNN op，缺口用 Metalium kernel 兜底。
Output结果回到 GGML / llama.cpp，必要时再转回 CPU 可读格式。
